# task-planner Skill

ジギョケイの認定基準（requirements）を元に、実装すべき機能のTDD（テスト駆動開発）用テストコードを自動生成します。

## 構成
- 入力: 認定基準（`src/core/certification_requirements.py` 等）、ユーザーのざっくりした要望
- 出力: `tests/unit/test_xxx.py` （失敗するテストコード）

## 手順
1. **プロトコル遵守の確認**: まず `core-protocol` Skill を参照し、現在の開発フェーズがTDDおよびドキュメント要件に沿っているかを確認する。
2. **ドメイン知識の参照**: `jigyokei-domain-expert` Skill を参照し、実装対象のセクションが満たすべき「認定審査基準（災害5要素、字数制限等）」を把握する。
3. **関連ファイルの調査**: `grep` や `find` を使用して、対象となるセクションの認定基準や既存の実装を確認する。
2. **要件の抽出**: 認定基準から「何が達成されるべきか」を具体的に抽出する。
3. **テストコードの生成**: TDDの「Red」フェーズとして、まず失敗するテストコードを作成する。
    - `pytest` を使用する形式で作成する。
    - 必要なモジュールのインポート、ダミーデータの設定を含める。
    - `assert` 文で期待される動作を定義する。
4. **GEMINI.mdの遵守**: `GEMINI.md` に定義された命名規則やコーディングスタイルに従う。

## 判断基準
- すでにテストがある場合: 既存のテストを拡張するか、新しいテストファイルを作成するかを判断する。
- 依存関係がある場合: 依存するコンポーネントのモック化を検討する。

## 実行例
「事業活動への影響セクションの自動生成ロジックを作りたい」
→ `task-planner` が起動
→ `certification_requirements.py` から `business_impact` の要件を確認
→ `tests/unit/test_business_impact.py` を生成
