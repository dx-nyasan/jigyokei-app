# ux-storyteller Skill

このSkillは、プロジェクトの進化を「技術的な機能」としてだけでなく、ユーザーが体験する「感動や価値」として記述し、ストックする役割を担います。

## 🎭 ストーリーテリングの原則 (UX Ideals)
1. **Empathy (共感)**: ユーザー（主に田中さんのような中小企業経営者）の悩みや不安を起点にする。
2. **Value Coding (価値の言語化)**: 単なる「機能名」ではなく、「それによって何が楽になったか」「どんなリスクが減ったか」を記述する。
3. **Continuity (継続性)**: `docs/DEVELOPMENT_HISTORY.md` と同期し、哲学の推移がどのようにUXに反映されたかを記録する。
4. **Guardianship (UX品質の守護)**: 開発成果が過去のストーリが示すユーザー体験を下回る（不便になる、複雑になる）場合、即座に警告を発する。

## 🛠️ 責務と手順

### 1. ストーリーの生成・更新
- 大規模な機能追加や、Phaseの完了時に起動する。
- `user_manual.md` および最新の実装を読み込み、象徴的なユーザー（田中さん）の体験がどのように変わったかをストーリー形式で追記する。
- 対象ファイル: `docs/USER_JOURNEY.md`

### 2. 体験の整合性チェック
- `jigyokei-domain-expert` と連携し、ストーリーの中でのAIの振る舞いが認定基準やドメインプロトコルから逸脱していないかを確認する。

### 3. ストック管理
- 過去のストーリーを損なうことなく、新しいScene（場面）を追加、あるいは既存のSceneをアップデートする。

## 🔄 他のSkillとの連携
- **technical-writer**: マニュアルを更新する際、ストーリーへの反映が必要かを検討し、必要であれば `ux-storyteller` を呼び出す。
- **core-protocol**: 全てのストーリーは、プロジェクトの「二大理念」に沿ったものでなければならない。

## 📝 実行例
「新しいAI監査機能をストーリーに追加して」
→ `ux-storyteller` が起動
→ 田中さんがダッシュボードで警告を受け、監査を実行し、納得して修正するプロセスを描写
→ `docs/USER_JOURNEY.md` を更新
